# Keeply React 노트 앱

> React·TypeScript·Redux Toolkit과 Express·SQLite로 구현한 과제 7 노트 앱입니다.

## 프로젝트 소개

Keeply는 로그인 사용자와 비회원이 서로 다른 저장 정책으로 사용할 수
있는 노트 앱입니다. 로그인 사용자의 노트는 로컬 SQLite에 영구 저장하고,
비회원 노트는 현재 브라우저 세션에만 저장합니다.

공식 필수 기술인 React, TypeScript, 상태관리 라이브러리를 사용했으며,
과제 도전 기능과 사용자 승인으로 추가된 인증·관리자 기능을 단계별로
구현했습니다. Google Antigravity IDE 대신 Codex를 사용했으며,
계획 확인, 작은 변경, 타입 검사와 테스트 과정을 동일하게 적용했습니다.

## 설치와 실행

루트에서 다음 명령을 실행합니다.

```bash
npm install
npm run dev
```

- 클라이언트: `http://127.0.0.1:5173`
- API: `http://127.0.0.1:3001`
- SQLite 파일: `server/data/keeply.db`

전체 검증 명령:

```bash
npm run typecheck
npm run lint
npm run test
npm run build
```

최초 관리자 계정은 비밀번호를 코드에 저장하지 않고 다음 명령으로
생성합니다.

```bash
npm run create-admin --workspace @keeply/server -- \
  --email admin@example.com \
  --name 관리자 \
  --password "10자 이상 비밀번호"
```

## 상태관리 구조

Redux store는 다음 원본 및 공용 화면 상태를 관리합니다.

| Slice | 역할 |
|---|---|
| `auth` | 사용자, 인증 상태, 로그인·로그아웃 요청 |
| `notes` | 회원·비회원 노트, 검색, 태그 필터, 상태, 정렬 |
| `tags` | 로그인 사용자의 태그 목록과 생성 상태 |
| `admin` | 관리자용 사용자 통계와 변경 요청 |
| `theme` | 라이트·다크 모드 |

제목·내용 같은 작성 중 입력, 열린 대화상자, 미리보기 탭처럼 특정
편집기에만 필요한 값은 React 로컬 상태로 관리합니다. 저장된 노트
원본은 컴포넌트 상태에 별도로 복제하지 않습니다.

## TypeScript 구조

- `client/src/types/note.ts`: `Note`, `Tag`, 정렬·상태 타입
- `client/src/types/auth.ts`: 사용자 역할과 인증 입력 타입
- `client/src/types/admin.ts`: 관리자 사용자 통계 타입
- `server/src/auth/types.ts`: 서버 인증 사용자와 세션 컨텍스트
- API 입력은 서버에서 Zod로 다시 검증
- `any`, `@ts-ignore` 없이 strict TypeScript 검사 통과

```text
client/src/
├── api/                  # 인증·노트·태그·관리자 API
├── components/

│   ├── auth/             # 로그인·회원가입
│   ├── common/           # 로더·테마 전환
│   ├── layout/           # 노트 작업공간
│   └── notes/            # 카드·편집기·태그·Markdown
├── hooks/                # 모달 포커스 관리
├── storage/              # 비회원 노트·테마 브라우저 저장
├── store/                # Redux slices와 typed hooks
├── types/                # 클라이언트 타입
└── utils/                # 색상 정규화·정렬

server/src/
├── auth/                 # 비밀번호·세션·사용자 조회

├── middleware/           # 인증 미들웨어

└── scripts/              # 최초 관리자 생성
```

## Codex 활용 과정

### 최초 계획에서 확인하거나 수정한 내용

- Antigravity 대신 Codex를 사용하되 계획 승인과 작은 작업 단위를 유지했습니다.
- 프론트엔드는 `client/`, 백엔드는 `server/`로 책임을 분리했습니다.
- 외부 DB 연동 대신 로컬 SQLite를 사용했습니다.
- 비회원 노트는 영구 저장 요구와 충돌하지 않도록 `sessionStorage`를 사용했습니다.
- 인증·관리자 기능은 클라이언트 표시가 아니라 서버 권한 검사로 구현했습니다.

### AI 초안에서 발견한 문제

- 자동 저장 효과에서 동기 상태 변경이 React Hooks 린트 규칙에 걸렸습니다.
- 다크 모드 리듀서 추가 중 기존 HMR 메모리 상태에 테마 값이 없어 일시 오류가 발생했습니다.
- Markdown HTML 차단 테스트가 한 HTML 블록의 뒤 텍스트까지 제거하는 실제 동작과 달랐습니다.
- 신규 자동 저장이 같은 노트를 반복 생성하지 않도록 작업 중 노트 ID 추적이 필요했습니다.

### 사람 검토 후 수정한 내용

- 로그인 화면 문구의 강제 줄바꿈을 반응형으로 유지했습니다.
- 비회원 경고 취소 시 편집기와 입력값을 유지했습니다.
- 로그아웃에서 쿠키 삭제뿐 아니라 서버 세션 revoke를 적용했습니다.
- 기본 색상 외에 네이티브 팔레트와 HEX 코드 입력을 추가했습니다.
- 접근성 보완 범위를 모달·키보드·모바일 탐색까지 확장했습니다.

## 오류 해결 기록

| 문제 | 원인 | 해결 | 재검증 |
|---|---|---|---|
| 자동 저장 린트 실패 | effect 본문의 동기 상태 변경 | 불필요한 대기 상태 제거 | lint 통과 |
| Markdown 안전성 테스트 실패 | 같은 HTML 블록 전체가 `skipHtml`로 제거됨 | 위험 HTML과 안전 문단을 분리한 테스트로 수정 | 관련 테스트 통과 |
| 다크 모드 HMR 일시 오류 | 리듀서 추가 전 store 메모리 상태 유지 | 개발 서버 재시작 | dev 서버 정상 |
| 사용자 색상 형식 차이 | 서버는 6자리 HEX만 허용 | 3·6자리 코드를 6자리 대문자로 정규화 | 색상 테스트 통과 |

## 한계와 추가 개선점

- 현재 SQLite는 단일 로컬 서버 프로세스를 기준으로 합니다.
- 비회원 노트는 브라우저 탭 종료 또는 세션 복원 정책에 따라 사라질 수 있습니다.
- 전체 브라우저 전수 검증과 화면 캡처는 제출 전에 별도로 수행해야 합니다.
- 관리자 계정 생성 시 명령행 비밀번호가 셸 기록에 남지 않도록 운영 환경에서는
  별도 비밀 입력 방식을 사용하는 편이 안전합니다.
