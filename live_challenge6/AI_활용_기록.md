# AI 활용 기록

## 1. AI 활용 개요

| 항목 | 내용 |
|---|---|
| 프로젝트 | React 쇼핑몰 |
| 사용 AI | OpenAI ChatGPT |
| 활용 목적 | 요구사항 분석, 프로젝트 구조 설계, TypeScript 코드 작성, 오류 분석, 기능 검토, 문서 작성 |
| 활용 방식 | 전체 코드를 한 번에 생성하지 않고 기능을 단계별로 나누어 요청 |
| 검토 방식 | 각 단계에서 TypeScript 검사, 빌드, 서버 실행 및 브라우저 기능 테스트를 직접 수행 |
| 최종 판단 | AI가 제안한 코드를 그대로 제출하지 않고 프로젝트 요구사항과 실제 실행 결과에 맞게 수정 |

AI는 구조 설계와 반복적인 코드 작성, 오류 원인 분석에 활용하였다. 각 기능을 적용한 뒤 직접 빌드하고 브라우저에서 실행했으며, 문제가 발견되면 해당 파일과 기능만 다시 수정하였다.

## 2. 프로젝트 구현 방식

과제 안내에서는 Firebase Authentication을 기본 인증 방식으로 제시했지만, 강사 안내에 따라 다른 인증 기술을 사용할 수 있어 Express, SQLite, bcrypt, JWT 기반 인증으로 대체하였다.

| 구분 | 과제 안내 기본 예시 | 실제 프로젝트 |
|---|---|---|
| 프론트엔드 | React | React + TypeScript + Vite |
| 전역 상태 | Redux 계열 | Redux Toolkit |
| 인증 | Firebase Authentication | Express + SQLite + bcrypt + JWT |
| 상품 데이터 | Fake Store API | Fake Store API + 로컬 mock 대체 |
| 재고 데이터 | 별도 요구 없음 | SQLite `product_inventory` 테이블 |
| 장바구니 저장 | 선택 사항 | 비회원 및 사용자별 LocalStorage |
| 결제 | 제외 범위 | 실제 PG 연동이 아닌 학습용 모의 결제 |
| 주문 내역 | 제외 범위 | SQLite 주문 및 주문 상품 기록 |
| 관리자 | 제외 범위 | 사용자 계정 및 상품 재고 관리 기능 확장 |

Firebase 대신 별도 인증 서버를 사용했지만 다음 인증 흐름은 동일하게 구현하였다.

- 회원가입
- 로그인
- 로그인 실패 안내
- 인증 초기 상태 확인
- 새로고침 후 로그인 상태 복원
- 로그인 사용자 정보 표시
- 로그아웃
- bcrypt 비밀번호 해시 저장
- JWT 발급 및 검증
- 환경변수를 통한 JWT 설정 관리
- 사용자 비밀번호 변경
- 관리자에 의한 임시 비밀번호 초기화
- 임시 비밀번호 로그인 후 비밀번호 변경 요구
- 일반 사용자 회원 탈퇴
- 관리자 계정 삭제 방지

모의 결제는 실제 카드 결제나 PG 서비스를 호출하지 않는다. 로그인 사용자의 장바구니 상품을 서버에서 다시 검증한 뒤 주문과 주문 상품을 SQLite에 저장하고 재고를 차감하는 학습용 기능이다.

## 5. 대표 프롬프트 및 수정 요청

### 5-1. 프로젝트 구조 설계

> React 쇼핑몰 과제를 TypeScript로 구현하려고 한다.  
> 클라이언트는 React와 Vite를 사용하고 서버는 Node, Express, TypeScript, SQLite를 사용한다.  
> 전체 작업을 한 번에 만들지 말고 하나씩 단계별로 진행하고, 각 단계마다 생성하거나 수정할 파일과 테스트 방법을 알려줘.  
> Git 설정은 제외하고 들여쓰기는 4칸을 사용해줘.

### 5-2. 상품 API 및 mock 대체

> Fake Store API에서 상품을 가져오고 실패할 경우 동일한 Product 타입의 로컬 mock 데이터를 사용하도록 해줘.  
> loading, error, empty 상태를 구분하고 상품 목록은 로그인하지 않아도 표시되게 해줘.

### 5-3. Redux 장바구니

> Redux Toolkit으로 장바구니를 구현해줘.  
> 필요한 기능은 상품 담기, 같은 상품 중복 추가 시 수량 증가, 수량 증가와 감소, 상품 삭제, 전체 삭제, 총액 계산, LocalStorage 저장이다.

### 5-4. 사용자별 장바구니 수정

> 로그아웃 했을 때 로컬 스토리지의 데이터가 그대로 남아서 로그아웃해도 장바구니 목록이 남아있는 문제가 있어.  
> 로그아웃하면 현재 화면의 장바구니는 초기화하고, 다시 로그인했을 때 해당 사용자가 담아뒀던 장바구니가 유지되도록 해줘.

첫 수정 이후 재로그인 시 장바구니가 보존되지 않아 다음과 같이 다시 요청하였다.

> 로그아웃 했다가 로그인해도 장바구니가 보존이 안 되어 있는데.

AI가 인증 사용자 ID를 즉시 저장 키로 사용하는 방식을 제안했지만, 인증 상태가 먼저 변경되면서 잘못된 키에 장바구니가 저장되는 문제가 발생하였다. 이후 cart 상태에 `ownerId`를 추가하고 사용자 전환 전에 기존 소유자의 장바구니를 저장하도록 수정하였다.

### 5-5. 로그인 후 화면 이동

> 로그인을 성공했는데 로그인 화면이 그대로 떠 있어.

로그인 성공 후 인증 상태만 변경되고 경로 이동이 없는 문제였다. 로그인 전 요청 경로를 유지하고, 로그인 완료 후 일반 사용자는 요청 화면으로, 관리자는 관리자 화면으로 이동하도록 수정하였다.

### 5-6. 재고 및 품절 정책

> 상품 상세 설명에 현재 재고 몇 개 남았는지를 표시하고, 수량이 10개 이하인 경우만 보여주고 싶다.  
> 품절 상품은 장바구니 담기를 회색 처리하고 재입고 알림 받기 기능도 추가하고 싶다.

최종 정책은 다음과 같다.

- 재고 11개 이상: 별도 수량 표시 없음
- 재고 1~10개: 정확한 잔여 수량 표시
- 재고 0개: 품절 표시
- 품절 상품 장바구니 버튼 비활성화
- 로그인 사용자만 재입고 알림 신청 가능
- 재입고 신청 상태를 사용자별 LocalStorage에 저장

### 5-7. 관리자 상품 재고 관리

> 관리자 계정은 상품 관리 탭도 하나 만들어서 상품 수량도 조절할 수 있도록 하면 좋을 것 같아.

초기 고정 재고를 SQLite로 이전하고 다음 API를 추가하였다.

- 일반 공개 재고 조회
- 관리자 상품별 재고 변경
- 재고 변경 시 수정 시간 저장
- 재고 0일 때 품절 처리

관리자 화면에서는 Fake Store API 상품 정보와 SQLite 재고를 결합하여 이미지, 상품명, 가격, 현재 재고, 변경 입력값을 표시하였다.

### 5-8. 비회원 로그인 전환

> 비회원이 결제하면 로그인 화면을 열고, 로그인 후에는 비회원 장바구니를 기존 계정 장바구니에 합칠지 덮어쓸지 선택하게 해줘.

적용한 정책:

- 신규 회원가입: 비회원 장바구니를 신규 계정으로 이전
- 기존 회원 로그인, 회원 장바구니가 비어 있음: 비회원 장바구니 이전
- 기존 회원 로그인, 양쪽 장바구니 모두 존재: 병합 또는 덮어쓰기 선택
- 관리자 계정: 쇼핑 장바구니 기능에서 제외

### 5-9. 장바구니 품절 검증

> 이미 장바구니에 넣어놨는데 품절되는 경우 장바구니에서 해당 상품은 품절된 상품입니다 하면서 구매 비활성화되도록 구현되어 있어?

기존 구현은 상품 목록과 상세에서만 재고를 확인하고 있어 장바구니에서도 공용 재고 API를 다시 조회하도록 수정하였다.

적용한 정책:

- 품절 상품 행 회색 처리
- 품절 안내 표시
- 수량 증가 버튼 비활성화
- 삭제 버튼 유지
- 장바구니 수량이 현재 재고보다 많으면 재고 부족 표시
- 품절 또는 재고 부족 상품이 있으면 결제 버튼 비활성화

### 5-10. 결제 직전 재고 검사

> 관리자 메뉴에서 변경된 재고 수량이 일반 사용자와 비로그인 사용자에게도 보이게 하고, 결제 직전에 다시 검사한 다음 결제 후 수량을 차감해줘.

서버 주문 트랜잭션에서 다음 순서로 처리하였다.

1. 로그인 사용자 확인
2. 주문 항목 형식 검증
3. 각 상품의 SQLite 재고 조회
4. 품절 및 재고 부족 검사
5. 조건부 재고 차감
6. 주문 생성
7. 주문 상품 생성
8. 전체 성공 시 커밋
9. 하나라도 실패하면 전체 롤백

### 5-11. 구매 내역 이미지 수정

> 구매 내역의 이미지가 깨지는데 이것도 Fake Store API로 불러오면 안 되나?

주문 당시의 상품명, 가격, 수량은 DB 값을 유지하고 이미지만 다음 순서로 표시하도록 수정하였다.

1. Fake Store API의 현재 상품 이미지
2. 주문 DB에 저장된 이미지
3. 로컬 대체 이미지

### 5-12. 주문 시간 수정

> 주문 시간이 UTC로 표시되는데 UTC+9로 보여줘.

SQLite `CURRENT_TIMESTAMP` 값은 UTC이므로 브라우저에서 UTC로 해석한 뒤 `Asia/Seoul` 시간대로 변환하였다. DB 원본 시간은 변경하지 않았다.

## 10. AI 활용에 대한 최종 평가

AI는 프로젝트 구조 설계, 반복적인 TypeScript 코드 작성, SQLite 테이블 및 API 작성, React UI 작성, 오류 원인 분석에 활용하였다. 그러나 생성된 코드를 그대로 사용하지 않고 다음 과정을 거쳤다.

1. 현재 프로젝트의 실제 파일 구조와 제안된 import 경로를 비교하였다.
2. 기존 타입과 새 타입이 일치하는지 확인하였다.
3. 서버에서 `npm run typecheck`를 실행하였다.
4. 클라이언트에서 `npm run build`를 실행하였다.
5. 서버와 클라이언트를 실행하여 브라우저에서 기능을 확인하였다.
6. 로그인, 로그아웃, 새로고침, 사용자 전환 순서를 반복 테스트하였다.
7. 비회원과 회원의 LocalStorage 저장 키를 직접 확인하였다.
8. 관리자 재고 변경값이 일반 사용자와 비회원 화면에 반영되는지 확인하였다.
9. 결제 성공과 실패 상황에서 주문과 재고 데이터가 일치하는지 확인하였다.
10. 문제가 발견되면 전체 프로젝트를 다시 작성하지 않고 관련 파일만 수정하였다.
11. 실제 인증 정보, JWT secret, 토큰, 계정 비밀번호는 기록에 포함하지 않았다.
12. 실제 카드 결제 기능이 아니라 학습용 모의 결제임을 문서에 명시하였다.

특히 사용자별 장바구니 보존 기능은 AI의 최초 제안만으로 정상 동작하지 않았다. 인증 상태가 바뀌는 시점과 장바구니 저장 시점이 충돌하여 기존 데이터가 잘못된 사용자 키에 저장되었고, 직접 테스트한 결과를 바탕으로 장바구니 소유자 상태와 저장 순서를 다시 설계하였다.

재고 기능도 처음에는 클라이언트의 고정 데이터였지만, 관리자 변경값을 모든 사용자에게 동일하게 보여주기 위해 SQLite 재고 테이블과 공용 API 구조로 변경하였다. 또한 클라이언트 화면의 재고만 신뢰하지 않고 서버가 결제 직전에 재고를 다시 확인하도록 보완하였다.

최종적으로 AI는 개발 속도를 높이는 도구로 사용하였으며, 실제 실행과 테스트 결과를 기준으로 코드를 채택, 수정 또는 거절하였다.
