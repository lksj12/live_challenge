# 오류 해결 기록

## 1. 기록 개요

| 항목 | 내용 |
|---|---|
| 프로젝트 | React 쇼핑몰 |
| 개발 환경 | React, TypeScript, Vite, Redux Toolkit, Express, SQLite |
| 인증 | Express, SQLite, bcrypt, JWT |
| 상품 데이터 | Fake Store API 및 로컬 mock 데이터 |
| 기록 범위 | TypeScript 설정, 인증, 장바구니, 재고, 관리자, 모의 결제, 구매 내역 |
| 해결 방식 | 오류 재현 → 원인 확인 → 최소 범위 수정 → 타입 검사 및 빌드 → 브라우저 재테스트 |
| 보안 원칙 | 실제 비밀번호, JWT 비밀값, 토큰, 개인 이메일은 기록하지 않음 |

이 프로젝트에서는 컴파일 오류뿐 아니라 콘솔에 오류가 나타나지 않는 상태 동기화 문제와 기능 흐름 오류도 함께 기록하였다.

오류가 발생했을 때 프로젝트 전체를 다시 작성하지 않고, 오류가 발생한 파일과 관련 기능만 수정하였다.

## 3. 상세 오류 해결 기록

### 3-1. TypeScript 입력 파일을 찾지 못하는 문제

#### 상황

Express 서버의 TypeScript 설정을 확인하기 위해 다음 명령을 실행하였다.

```bash
npx tsc --noEmit
```

#### 오류 메시지

```text
error TS18003: No inputs were found in config file
```

#### 원인

- `tsconfig.json`의 `include` 경로가 실제 소스 폴더와 일치하지 않았다.
- 서버 시작 파일인 `index.ts`가 `server/src`가 아니라 `server` 루트에 있었다.
- 설정 파일 이름과 실제 프로젝트 구조를 함께 확인할 필요가 있었다.

#### 수정

`tsconfig.json`의 검사 대상을 다음과 같이 변경하였다.

```json
{
    "include": [
        "src/**/*.ts"
    ]
}
```

서버 시작 파일도 다음 위치로 이동하였다.

```text
server/src/index.ts
```

#### 재테스트

```bash
npx tsc --noEmit
```

#### 결과

TypeScript가 서버의 소스 파일을 정상적으로 찾았으며 타입 검사를 통과하였다.

### 3-2. 상품 목록 페이지 파일 누락 문제

#### 상황

`App.tsx`에서 홈 경로에 `ProductPage`를 연결한 뒤 클라이언트를 빌드하였다.

#### 오류 메시지

```text
ProductPage 모듈을 찾을 수 없음
```

#### 원인

`App.tsx`에는 다음 import가 있었지만 실제 파일 생성이 누락되어 있었다.

```ts
import ProductPage from './pages/ProductPage';
```

#### 수정

다음 파일을 생성하였다.

```text
client/src/pages/ProductPage.tsx
```

상품 Redux 상태와 loading, error, empty, Fake Store API 상품, mock 대체 상품, 검색, 카테고리 필터를 연결하였다.

#### 재테스트

```bash
cd client
npm run build
npm run dev
```

#### 결과

홈 경로에서 상품 목록이 정상적으로 표시되었고 검색과 카테고리 필터도 동작하였다.

### 3-3. 로그인 성공 후 로그인 화면이 유지되는 문제

#### 상황

올바른 이메일과 비밀번호를 입력하여 로그인하였다.

#### 발생 현상

- 로그인 요청은 성공하였다.
- 헤더의 사용자 정보는 갱신되었다.
- 그러나 화면 경로가 `/auth`에 그대로 남아 로그인 화면이 계속 표시되었다.

#### 원인

로그인 성공 후 Redux 인증 상태만 변경하고 React Router 경로를 이동하지 않았다.

#### 수정

로그인 전에 사용자가 이동하려던 경로를 확인하고, 로그인 성공 후 역할과 비밀번호 상태에 따라 이동하도록 수정하였다.

```text
일반 사용자 → 로그인 전 요청 화면
관리자 → /admin
임시 비밀번호 사용자 → /account
```

회원가입을 완료한 경우에도 자동 로그인한 뒤 이동하도록 변경하였다.

#### 재테스트

- 일반 사용자 로그인
- 관리자 로그인
- 장바구니에서 로그인 화면으로 이동 후 로그인
- 임시 비밀번호 사용자 로그인

#### 결과

로그인 성공 후 로그인 화면이 사라지고 사용자 역할과 이전 요청에 맞는 화면으로 이동하였다.

### 3-4. 사용자별 장바구니가 보존되지 않는 문제

#### 상황

1. 사용자 A로 로그인한다.
2. 상품을 장바구니에 담는다.
3. 로그아웃한다.
4. 사용자 A로 다시 로그인한다.
5. 사용자 B로도 로그인하여 장바구니를 비교한다.

#### 발생 현상

- 로그아웃 후 이전 사용자의 장바구니가 화면에 남는 경우가 있었다.
- 같은 사용자로 다시 로그인했을 때 저장했던 장바구니가 복원되지 않았다.
- 사용자 A와 사용자 B의 장바구니가 잘못된 저장소에 저장될 가능성이 있었다.

#### 원인

로그아웃 과정에서 인증 사용자 정보가 먼저 `null`로 변경될 수 있어 기존 사용자 장바구니가 비회원 저장소에 저장되거나 빈 값으로 덮어쓰일 수 있었다.

#### 수정

cart 상태에 다음 정보를 추가하였다.

```ts
interface CartState {
    ownerId: number | null;
    initialized: boolean;
    items: CartItem[];
}
```

사용자 상태가 바뀔 때 다음 순서로 처리하였다.

1. 현재 cart의 `ownerId`를 확인한다.
2. 현재 장바구니를 기존 소유자의 저장소에 저장한다.
3. 새로운 사용자의 저장 장바구니를 읽는다.
4. 비회원 장바구니가 있으면 이전 정책에 따라 처리한다.
5. cart의 `ownerId`를 새 사용자 ID로 변경한다.

#### 결과

- 같은 사용자의 장바구니가 재로그인 후 복원되었다.
- 서로 다른 사용자의 장바구니가 섞이지 않았다.
- 비회원 장바구니와 회원 장바구니를 별도로 처리할 수 있게 되었다.

### 3-5. 로그아웃 후 화면에 기존 정보가 남는 문제

#### 발생 현상

토큰과 인증 상태는 제거되었지만 일부 화면에서 이전 사용자 이름이나 장바구니 내용이 남았다.

#### 원인

인증 상태 변경, 장바구니 저장, React 렌더링이 순차적으로 실행되면서 일부 컴포넌트가 이전 렌더링 상태를 잠시 유지하였다.

#### 수정

```ts
function handleLogout(): void {
    dispatch(logout());
    window.location.replace('/');
}
```

#### 결과

로그아웃 직후 홈 화면이 다시 로드되었고 이전 사용자 정보가 제거되었다.

### 3-6. inventoryController TypeScript 문법 오류

#### 오류 메시지

```text
ERROR: Expected ";" but found "Partial"
```

#### 원인

```ts
const body =
    request.body
    as Partial<UpdateProductStockRequestBody>;
```

`request.body`와 TypeScript 타입 단언 연산자인 `as`를 분리하여 줄바꿈하였다.

#### 수정

```ts
const body =
    request.body as Partial<UpdateProductStockRequestBody>;
```

#### 결과

서버 TypeScript 검사와 개발 서버 실행이 정상적으로 완료되었다.

### 3-7. RestockNotification null 타입 오류

#### 오류 메시지

```text
'currentUser' is possibly 'null'
Argument of type 'number | null' is not assignable to parameter of type 'number'
```

#### 원인

이벤트 함수가 나중에 실행될 수 있으므로 TypeScript는 기존 null 검사가 계속 유효하다고 보장하지 않았다.

#### 수정

```ts
const authenticatedUserId =
    currentUser.id;

const notifiedProductId =
    restockedProductId;
```

이벤트 함수에서는 고정한 지역 상수를 사용하였다.

#### 결과

null 관련 TypeScript 오류가 해결되고 클라이언트 빌드가 통과하였다.

### 3-8. 관리자 재고가 일반 사용자에게 반영되지 않는 문제

#### 발생 현상

관리자 상품 관리 화면에서는 변경된 재고가 표시되었지만 일반 사용자와 비회원 화면에서는 이전 수량이 표시되었다.

#### 원인

관리자 화면은 SQLite 재고를 사용했지만 일반 상품 목록, 상세 화면, 장바구니는 기존 TypeScript 고정 재고를 사용하고 있었다.

#### 수정

공용 재고 조회 API를 추가하였다.

```text
GET /api/inventory
```

상품 목록, 상품 상세, 장바구니, 재입고 알림, 관리자 상품 관리가 같은 Redux inventory 상태를 사용하도록 수정하였다.

#### 결과

관리자, 일반 사용자, 비로그인 사용자가 모두 동일한 SQLite 재고를 확인하게 되었다.

### 3-9. 장바구니에 담긴 상품의 품절 상태 미반영 문제

#### 발생 현상

상품 목록에서는 품절로 표시되었지만 장바구니에서는 수량 증가와 결제를 시도할 수 있었다.

#### 원인

장바구니가 LocalStorage에 저장된 상품 정보만 사용하고 현재 SQLite 재고를 다시 조회하지 않았다.

#### 수정

```ts
const stock =
    stockByProductId[item.productId] ?? 0;

const soldOut =
    stock === 0;

const insufficientStock =
    stock > 0
    && item.quantity > stock;
```

품절 안내, 재고 부족 안내, 수량 증가 제한, 결제 버튼 비활성화를 적용하였다.

#### 결과

품절 또는 재고 부족 상품이 있으면 사용자에게 원인을 안내하고 결제를 제한하였다.

---

### 3-10. 결제 버튼이 계속 비활성화된 문제

#### 발생 현상

재고가 충분하고 로그인 상태였지만 버튼에는 `결제 기능 준비 중`이 표시되고 클릭할 수 없었다.

#### 원인

서버 주문 기능은 구현되었지만 프론트에 초기 임시 버튼이 남아 있었다.

```tsx
<button
    type="button"
    className="checkout-button"
    disabled
>
    결제 기능 준비 중
</button>
```

#### 수정

실제 주문 API를 호출하는 `handleCheckout()`을 연결하고 다음 조건일 때만 버튼을 비활성화하였다.

```tsx
disabled={
    hasUnavailableItem
    || isCheckingOut
}
```

#### 결과

재고가 충분한 경우 `결제하기` 버튼이 활성화되고 주문 API가 정상적으로 호출되었다.

#### 오류 메시지

```text

```

#### 원인

TypeScript에서는 catch 블록의 오류 변수를 기본적으로 `unknown`으로 취급한다.

#### 수정

```ts
const message =
    error instanceof ApiRequestError
        ? error.message
        : error instanceof Error
            ? error.message
            : '결제 처리 중 오류가 발생했습니다.';
```

#### 결과

TypeScript 타입 오류가 해결되었고 서버 응답 오류와 일반 JavaScript 오류를 각각 표시할 수 있게 되었다.

### 3-13. 구매 내역 상품 이미지가 깨지는 문제

#### 발생 현상

구매 내역에서 상품 이미지 대신 깨진 이미지 아이콘이 표시되었다.

#### 원인

주문 DB에 저장된 외부 이미지 주소만 사용하고 있었다.

#### 수정

이미지 사용 순서를 다음과 같이 정하였다.

1. Fake Store API에서 다시 조회한 이미지
2. 주문 DB에 저장된 이미지
3. 로컬 대체 이미지

상품명, 가격, 카테고리, 수량은 주문 당시 DB 값을 계속 사용하였다.

#### 결과

구매 내역 이미지가 정상적으로 표시되었고 외부 이미지 오류가 발생해도 대체 이미지가 표시되었다.

### 3-14. API 상품 배지 제거 후 빌드 오류

#### 오류 메시지

```text
Cannot find name 'selectProductSource'
Type 'unknown' is not assignable to type 'ReactNode'
```

#### 원인

자동 수정 스크립트가 `selectProductSource` import는 삭제했지만 selector 변수와 배지 JSX를 남겨두었다.

#### 수정

다음 항목을 모두 제거하였다.

- `selectProductSource` import
- `source` selector 변수
- `source`를 사용하는 배지 JSX

#### 결과

`API 상품` 배지가 제거되고 클라이언트 빌드가 정상적으로 완료되었다.

### 3-15. SQLite 주문 시간이 한국 시간과 다른 문제

#### 발생 현상

구매 내역 주문 시간이 실제 한국 시간보다 9시간 차이 났다.

#### 원인

SQLite의 `CURRENT_TIMESTAMP`는 UTC 기준이고 저장 문자열에는 시간대 정보가 없었다.

#### 수정

시간대 정보가 없는 SQLite 문자열을 UTC로 해석한 뒤 화면에서는 `Asia/Seoul` 시간대로 포맷하였다.

```ts
new Intl.DateTimeFormat(
    'ko-KR',
    {
        timeZone: 'Asia/Seoul',
        year: 'numeric',
        month: '2-digit',
        day: '2-digit',
        hour: '2-digit',
        minute: '2-digit',
    },
).format(date);
```

#### 결과

DB에는 UTC 시간이 유지되고 구매 내역 화면에서는 한국 시간으로 정확하게 표시되었다.

---

### 3-16. 결제 순간 실제 재고가 달라질 수 있는 문제

#### 문제점

브라우저에서 재고를 확인한 뒤 다른 사용자가 먼저 구매하거나 관리자가 재고를 변경하면 화면의 값이 최신 상태가 아닐 수 있었다.

#### 수정

서버 주문 트랜잭션에서 SQLite 재고를 다시 조회하고 조건부 UPDATE를 사용하였다.

```sql
UPDATE product_inventory
SET
    stock = stock - ?,
    updated_at = CURRENT_TIMESTAMP
WHERE
    product_id = ?
    AND stock >= ?
```

재고 부족 시 `409 Conflict` 응답을 반환하였다.

#### 결과

결제 순간 서버가 최신 재고를 기준으로 주문을 허용하거나 거부하게 되었다.

---

### 3-17. 주문 생성과 재고 차감의 데이터 정합성 문제

#### 잠재적 문제

주문 생성과 재고 차감을 별도로 실행하면 한쪽만 성공할 수 있었다.

#### 수정

다음 작업을 하나의 SQLite 트랜잭션으로 처리하였다.

1. 모든 주문 상품의 현재 재고 검사
2. 모든 주문 상품의 재고 조건부 차감
3. orders 테이블에 주문 생성
4. order_items 테이블에 상품 저장
5. 전체 성공 시 commit
6. 하나라도 실패하면 rollback

#### 결과

주문과 재고가 함께 성공하거나 함께 취소되었다. 결제 실패 시 주문이 생성되지 않고 재고도 줄어들지 않았다.

## 5. 예방 체크리스트

- TypeScript 파일은 `tsconfig.json`의 `include` 범위 안에 둔다.
- import를 추가하기 전에 대상 파일과 export 위치를 확인한다.
- AI가 제안한 import 경로가 현재 프로젝트 구조와 일치하는지 확인한다.
- catch 오류 변수는 사용 전에 타입을 확인한다.
- React 이벤트 함수에서 nullable 값을 사용할 때 지역 상수로 고정한다.
- 자동 수정 스크립트로 UI를 제거할 때 import, 변수, JSX를 모두 확인한다.
- 인증 사용자와 장바구니 소유자를 별도 상태로 관리한다.
- 사용자 전환 전에 기존 소유자의 장바구니를 저장한다.
- 비회원과 회원의 LocalStorage 키를 분리한다.
- 상품 재고의 기준 데이터는 SQLite 한 곳에서 관리한다.
- 상품 목록, 상세, 장바구니, 관리자 화면이 같은 재고 API를 사용하도록 한다.
- 장바구니에서는 저장 당시 재고가 아닌 현재 재고를 다시 확인한다.
- 결제 직전 서버에서 실제 재고를 다시 확인한다.
- 재고 차감 시 `stock >= quantity` 조건을 사용한다.
- 주문 생성과 재고 차감은 하나의 트랜잭션으로 처리한다.
- 외부 이미지 URL은 실패할 수 있으므로 fallback 이미지를 제공한다.
- SQLite의 `CURRENT_TIMESTAMP`가 UTC임을 고려한다.
- DB 시간은 UTC로 유지하고 화면에서 필요한 시간대로 변환한다.
- 실제 비밀번호, JWT secret, 토큰, 개인 이메일은 오류 기록에 포함하지 않는다.
- 변경 후 서버 `npm run typecheck`와 클라이언트 `npm run build`를 실행한다.
- 빌드 성공 후 브라우저에서 실제 기능을 다시 테스트한다.

## 6. 최종 정리

이번 프로젝트에서는 TypeScript 컴파일 오류, 모듈 경로 오류, React 상태 동기화 문제, LocalStorage 저장 순서 문제, 재고 데이터 중복 문제, 결제 시점의 데이터 정합성 문제를 해결하였다.

사용자별 장바구니 문제는 LocalStorage 키를 분리하는 것만으로 해결되지 않았다. 인증 상태가 변경되는 시점과 장바구니가 저장되는 시점을 함께 고려하여 cart 상태에 실제 소유자 정보를 추가하였다.

상품 재고도 처음에는 클라이언트의 고정 데이터로 관리했지만, 관리자 변경값을 일반 사용자와 비로그인 사용자에게 동일하게 보여주기 위해 SQLite 재고 테이블과 공용 API 구조로 변경하였다.

브라우저 화면에 표시된 재고는 결제 시점의 최신 재고를 보장하지 않으므로 서버에서 결제 직전에 재고를 다시 확인하였다. 주문 생성과 재고 차감은 하나의 SQLite 트랜잭션으로 처리하여 한쪽만 성공하는 데이터 불일치를 방지하였다.

AI가 생성한 코드는 현재 프로젝트의 파일 구조, 타입, export 위치와 일치하지 않는 경우가 있었으며, 실제 빌드 오류와 기능 테스트 결과를 기준으로 수정하였다.

최종적으로 서버 TypeScript 검사, 클라이언트 프로덕션 빌드, 로그인과 사용자 전환, 장바구니 복원, 관리자 재고 변경, 모의 결제, 재고 차감, 구매 내역 표시를 직접 확인하였다.
