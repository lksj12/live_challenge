# Keeply Notes 에이전트 작업 가이드

## 프로젝트 목표

React와 TypeScript로 노트 앱을 만들고 Redux Toolkit으로 클라이언트
상태를 관리한다. 로그인한 사용자의 데이터는 로컬 SQLite에 저장하고,
비회원 노트는 브라우저 세션에만 저장한다. 관리자 기능, 취소 가능한
확인 대화상자, 서버에서 폐기되는 로그인 세션과 과제의 도전 기능을
모두 구현한다.

## 기준 문서

다음 순서로 요구사항과 작업 범위를 판단한다.

1. `과제7_React노트앱_과제안내.ipynb`
2. `docs/spec.md`
3. `docs/plan.md`
4. `AGENT_GUIDE.md`와 `.agents/rules/project.md`
5. 현재 프로젝트 코드와 설정

문서와 코드가 충돌하면 성공 결과를 추정하지 않고 실제 코드와 검증
결과를 다시 확인한다.

## 디렉터리 책임

- `client/`: React 화면, Redux Store, 브라우저 저장소, 프론트 테스트
- `server/`: SQLite, 인증, 세션, API, 백엔드 테스트
- `.agents/rules/`: 프로젝트 작업 규칙
- 루트: 공통 실행 설정, 과제 안내, 작업 기록과 제출 문서

프론트엔드 코드를 `server/`에 두거나 백엔드 코드를 `client/`에 두지
않는다. 공통 타입이 필요해지기 전까지 별도 공유 패키지를 만들지 않는다.

## 필수 작업 절차

1. 작업 전에 이 문서와 `.agents/rules/project.md`를 읽는다.
2. 한 번에 하나의 작은 기능에 대한 구현 계획을 작성하고 사용자 승인을 받는다.
3. 승인된 기능에 필요한 파일만 수정한다.
4. TypeScript 검사, 관련 테스트, 린트와 빌드를 실행한다.
5. 변경 파일, 변경 이유, 검증 결과, 실패와 남은 위험을 보고한다.

## 기술 규칙

- 모든 앱 코드는 TypeScript로 작성하고 엄격한 타입 검사를 유지한다.
- `any`를 사용하지 않으며 입력 데이터는 런타임에서도 검증한다.
- 노트 원본 데이터는 Redux Store에서 관리하고 컴포넌트 상태에 중복 저장하지 않는다.
- 폼 입력, 대화상자, 일시적인 안내 메시지는 컴포넌트 상태로 둘 수 있다.
- 로그인한 사용자의 영구 데이터는 SQLite를 단일 기준으로 사용한다.
- 비회원 노트는 `sessionStorage`에만 저장하고 서버로 전송하지 않는다.
- 로딩, 빈 상태, 성공과 실패 상태를 화면과 보조 기술에 명확히 전달한다.

## 인증과 권한 규칙

- 인증, 노트 소유권, 관리자 권한은 모든 요청에서 서버가 검사한다.
- 클라이언트가 보낸 사용자 ID나 역할을 신뢰하지 않는다.
- 비밀번호는 Argon2id 해시만 저장하며 원문 비밀번호를 기록하거나 노출하지 않는다.
- 로그인 토큰은 예측 불가능한 불투명 토큰을 사용하고 서버에는 해시만 저장한다.
- 로그아웃, 비밀번호 초기화, 계정 비활성화 시 관련 세션을 서버에서 폐기한다.
- 관리자는 사용자 비밀번호와 노트 내용을 열람할 수 없다.
- 비밀정보, 세션 토큰, 실제 개인정보를 코드·로그·테스트 데이터에 넣지 않는다.

## 제품 범위

- 외부 데이터베이스, 외부 OAuth, 실시간 협업과 자동 배포는 사용하지 않는다.
- 로그인 사용자는 전체 노트 기능을 사용하고 비회원은 기본 노트 기능만 사용한다.
- 비회원은 태그를 생성하거나 지정할 수 없다.
- 비회원 저장 확인을 취소하면 작성 중인 입력값과 편집 창을 그대로 유지한다.
- 과제 안내 노트북을 수정하거나 삭제하지 않는다.

## 변경 제한

- 승인 없이 파괴적인 Git 명령, push, 대량 삭제, 전체 구조 변경을 하지 않는다.
- 승인된 작업에 필요하지 않은 패키지를 추가하거나 전체 의존성을 갱신하지 않는다.
- 관련 없는 리팩터링, 파일명 일괄 변경과 스타일 전체 교체를 하지 않는다.

## 검증 명령

루트에서 다음 명령을 실행한다.

- `npm run typecheck`
- `npm run lint`
- `npm run test`
- `npm run build`

현재 루트 검증 명령은 `client/`와 `server/`를 모두 검사한다. 브라우저
검증은 기능 계획에 포함되거나 사용자가 요청한 경우 수행한다.
