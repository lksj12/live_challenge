# 과제5 AI 결과 검토표

> 목적: AI가 제안한 코드와 설계를 사람이 어떤 기준으로 검증하고 수정했는지 기록합니다.  
> 최종 문서 수정일: 2026-07-11

---

## 1. 전체 검토 요약

| 항목 | 최종 내용 |
|---|---|
| 프로젝트 | 학습 기록 관리 앱 |
| 구조 | React + Express + SQLite |
| CRUD | 사용자별 학습 기록 조회·추가·수정·삭제 |
| 인증 | JWT + bcryptjs |
| 로그아웃 | JWT `jti` revoke |
| 역할 | admin/user |
| 관리자 생성 | 공개 가입이 아닌 로컬 스크립트 |
| 프론트 구조 | API, components, hooks, pages, utils 분리 |
| 추가 기능 | 닉네임, 한국 시간 표시, CORS 제한 |
| 현재 상태 | 기능 구현 완료 단계, 제출 전 최종 통합 테스트·캡처·링크 입력 필요 |

## 2. 요구사항 검토

| 검토 항목 | AI 제안 | 사람이 검토한 기준 | 최종 결정 | 상태 |
|---|---|---|---|:---:|
| CRUD 주제 | 여러 CRUD 앱 추천 | 과제 맥락과 4개 기능이 명확한가 | 학습 기록 관리 앱 | 통과 |
| 저장 방식 | LocalStorage 또는 DB | 제출 폴더 재현성과 다중 접속이 가능한가 | Express + SQLite | 통과 |
| 시간 저장 | 정수 시간 | 30분, 1시간 20분 등을 정확히 표현 가능한가 | 총 분으로 저장 | 통과 |
| 사용자별 데이터 | user_id 조건 | 타인 기록 접근을 막는가 | 모든 조회·수정·삭제에 user_id 적용 | 통과 |
| 역할 분리 | admin/user | 역할별 기능이 중복되지 않는가 | admin은 계정, user는 학습 기록 | 통과 |
| 프론트 구조 | App 중심 구현 | 인증 기능 추가 후 유지보수 가능한가 | 페이지·훅·API·컴포넌트 분리 | 통과 |
| 닉네임 | 회원가입 필드 추가 | 헤더·관리자 목록에 필요한가 | 일반 사용자는 입력, 관리자는 Admin | 통과 |
| 표시 시간 | 원본 문자열 표시 | UTC와 한국 시간이 일치하는가 | Asia/Seoul 변환 | 통과 |

## 3. CRUD 코드 검토

## 4. 인증·보안 검토

| 항목 | 초기 AI 제안 또는 상태 | 검토 기준 | 최종 보완 | 판단 |
|---|---|---|---|:---:|
| 비밀번호 | 해싱 필요 | 평문 금지 | bcryptjs | 통과 |
| JWT 검증 | 서명·만료 확인 | 보호 API가 토큰을 검사하는가 | requireAuth | 통과 |
| 권한 검증 | role 사용 | 프론트 role을 신뢰하지 않는가 | requireAdmin/requireUser | 통과 |
| 관리자 가입 | ADMIN_EMAIL 또는 가입 코드 검토 | 공개 API로 admin 생성 가능성이 없는가 | create-admin 스크립트 | 통과 |
| 로그아웃 | 로컬 토큰 삭제 | 복사한 토큰을 다시 쓸 수 있는가 | jti blacklist | 통과 |
| CORS | 모든 origin 허용 | 개발 프론트 주소만 허용하는가 | CLIENT_URL 제한 | 통과 |
| 토큰 저장 | localStorage | XSS 위험을 인지했는가 | 한계 문서화 | 조건부 통과 |
| 삭제 사용자 토큰 | 별도 세션 없음 | 삭제 후 기존 토큰 전체 무효화 가능한가 | 미구현 한계 기록 | 보완 필요 |
| 요청 제한 | 없음 | 무차별 로그인 시도 방지 | 미구현 | 향후 개선 |
| HTTPS | 로컬 HTTP | 전송 구간 암호화 | 배포 시 적용 | 향후 개선 |

## 6. AI 결과를 그대로 쓰지 않고 수정한 사례

| 사례 | AI 또는 초기 구현 | 사람이 판단한 문제 | 최종 수정 |
|---|---|---|---|
| CRUD 주제 | 일반 todo 가능 | 과제 차별성과 기록 목적 부족 | 학습 기록 관리 |
| 시간 입력 | 정수 시간 | 분 단위 표현 불가 | 시간/분 입력 + 총 분 저장 |
| 수정폼 초기화 | useEffect에서 여러 setState | lint 경고와 불필요한 렌더 가능성 | `key`로 재마운트 |
| 관리자 가입 | 이메일 또는 가입 코드 | 공개 API 악용 가능 | 관리자 생성 스크립트 |
| 로그아웃 | 클라이언트 토큰 삭제 | 기존 토큰이 만료 전까지 유효 | jti revoke |
| 프론트 구조 | App에 상태·목록·폼 집중 | 인증·관리자 기능 추가 시 비대화 | 기능별 모듈 분리 |
| 시간 표시 | SQLite 문자열 그대로 표시 | UTC가 한국 시간보다 9시간 느림 | Intl + Asia/Seoul |
| 사용자 표시 | 이메일과 role | 사용자 친화성 부족 | 닉네임과 이메일 표시 |
| 인증 모드 전환 | 일부 입력값 유지 | 다른 모드의 값이 남음 | 모든 입력값 초기화 |

## 7. API 검토표

|---|---|---|:---:|

| `DELETE /api/studies/:id` | user | id + user_id | 통과 |

## 8. 사람이 직접 결정한 사항

| 결정 | 이유 |
|---|---|
| SQLite 사용 | 제출 폴더만으로 재현하고 동일 서버 사용자 간 데이터를 공유하기 위해 |
| DB 재생성 | 개발 중 스키마가 크게 바뀌어 마이그레이션보다 새 DB가 단순했기 때문 |
| 관리자와 사용자 역할 분리 | 과제 기능과 권한을 명확히 하기 위해 |
| 관리자 닉네임 Admin 고정 | 관리자 계정의 표시를 일관되게 하기 위해 |
| 일반 사용자 Admin 닉네임 금지 | 관리자 표시와 혼동을 피하기 위해 |
| DB UTC 유지 | 서버 환경과 관계없이 저장 시간을 일관되게 유지하기 위해 |
| 한국 시간은 프론트에서 변환 | 표시 지역에 맞추되 DB 원본은 UTC로 유지하기 위해 |
| Supabase 미사용 | 로컬 재현성과 SQLite 학습 목적을 우선했기 때문 |

## 10. 최종 판단

- 핵심 CRUD, 인증, 역할 분리, 관리자 기능, 프론트 구조 분리는 과제 요구 범위를 충족합니다.
- AI가 제안한 코드는 단계별 실행과 사용자 검토를 거쳐 구조·보안·사용성을 수정했습니다.
- 최종 제출 전에는 위 체크리스트, 화면 캡처, README 링크를 직접 완성해야 합니다.
- 실제 서비스 수준에서는 사용자별 모든 토큰 폐기, HttpOnly Refresh Token, rate limiting, HTTPS 등을 추가해야 합니다.
